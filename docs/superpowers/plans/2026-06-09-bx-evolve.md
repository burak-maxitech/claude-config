# /bx:evolve Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the 11th `bx` skill, `/bx:evolve` — an upstream-watch audit that researches Anthropic changes (changelog / official docs / community) since a committed watermark, gates findings against a capability inventory of the plugin, reports with mandatory citations, and applies edits behind a per-finding diff gate.

**Architecture:** Orchestrator SKILL.md in the bx house idiom (Step 0 context → parallel Sonnet agent dispatch → consolidate/gate → report → mode tail), three new least-privilege Sonnet agents (one per research lane), six reference files, and a committed state file (`docs/upstream/state.json`) holding the watermark + decision log. Spec: `docs/superpowers/specs/2026-06-09-bx-evolve-design.md` — read it before starting; it is the contract.

**Tech Stack:** Markdown skill/agent files (Claude Code plugin format), JSON state, `gh` CLI + WebFetch/WebSearch for research.

**Conventions that bind every task** (from this repo's S42/S43/S45 lessons):
- Every command a file tells the model to run MUST appear in that file's `allowed-tools` (skill) or `tools` (agent) frontmatter.
- Dispatch names the dedicated agents via Task-tool `subagent_type` (never "spawn a generic subagent").
- No `${CLAUDE_SKILL_DIR}` and no pseudo-paths — reference sibling files as `references/<name>.md`; cross-skill files as `../<skill>/references/<name>.md` relative to the skill base directory.
- No assumption that shell state (env vars, cwd) persists across Bash calls.
- Tier-2 (community) findings are advisory-only and never `--fix`-eligible — this rule must appear in every file that touches findings.

- [ ] **Step 1: Write the file.** Structure (mirror `bx/skills/tests/references/scan-coverage.md` style — "Inputs you receive in your task prompt" then method then output shape):
  - **Inputs:** watermark (`last_changelog_version` or null), capability inventory, pain-point list, tier definitions.
  - **Method:** `gh release list --repo anthropics/claude-code --limit 50` to enumerate releases newer than the watermark; for each, `gh release view <tag> --repo anthropics/claude-code --json body,tagName,publishedAt`. Each candidate delta is checked against the capability inventory; no intersection and no pain-point match → discard silently.
  - **Output shape:** the finding schema verbatim from the spec (`finding_id`, `class: breakage|best_practice|opportunity`, `tier: official`, `severity`, `certainty`, `affected_files`, `upstream_delta`, `proposed_edit`, `citation`, `source_content_hash`), plus a footer addendum: `releases_scanned: <n>`, `newest_version_seen: <tag>`, `lane_status: ok|degraded|unavailable`.

  - **Section 3 — Advisory (community):** clearly badged, "consider" phrasing, never offered to `--fix`.
  - **Section 4 — Carried-forward open findings:** findings still `open` from previous runs, with their original report date.
  - **Footer disclosure:** lanes run + `lane_status` each, pages/releases fetched vs caps, decision-log filters applied (`<n> rejected findings suppressed, source unchanged`), watermark old → new, count of `open` entries written.

- [ ] **Step 2: Write `fix-mode-evolve.md`.** Mirror `bx/skills/arch/references/fix-mode.md`'s gate flow, adapted:
  - Eligibility: `tier: official` AND `decision != rejected` AND the proposed edit is a doc/skill-file text change inside this repo. Advisory findings never appear.
  - Per-finding gate: show `upstream_delta` + citation, the full `affected_files` list, one combined diff covering ALL affected files (sibling echoes applied together — partial application reintroduces drift), then `Apply? [y / n / skip / abort]`.
  - Verdict handling: `y` → apply via Edit to every affected file, decision `applied`; `n` → `rejected`; `skip` → stays `open`; `abort` → stop, applied edits stay, remaining stay `open`.
  - After the pass: rewrite `docs/upstream/state.json` (per `state-schema.md`), print summary (applied/rejected/skipped counts), remind `/plugin update bx` + `/reload-plugins`, and recommend the S42 content-review treatment for any skill that received non-trivial edits.
