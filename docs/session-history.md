Iterated on the name (`/whats-next` and `/suggest-flow` rejected as implying business tasks; user proposed `-advice` suffix to signal "report only, never invokes"). Final name: `/code-health-advice` (18 chars; shorter than `/architecture-review` so length is fine; the suffix earns its keep by disambiguating against `/code-review` and `/code-cleanup` which both have `--fix`).
- **Three deliberate non-features for `/code-health-advice`.** (1) Never invokes other skills — output is text only, user runs the suggested flow themselves. (2) Never edits files — there is no `--fix` mode and there will not be one. (3) Exactly one Recommended flow + one Alternative flow per report — more is noise. Codified all three in `SKILL.md` Step 3 rules and Step 4 (explicit "stop, do not ask 'want me to run X for you?'").
- **Five state buckets** (full catalog in `references/state-buckets.md`): A. Pre-commit cleanup (uncommitted changes on feature branch), B. Pre-merge verification (clean tree + open PR), C. Post-ship audit (clean tree on main, recent ship), D. Orient + audit (long since last commit, or no `CLAUDE.md`), E. Ambient improvement (clean tree, no urgent task). Each has a recommended flow and an alternative flow with a one-line "use this if…" qualifier (e.g., Bucket A's alternative: *"if this touches auth/payments/migrations"* → adds `--security` + `/ultrareview`).
- **Tie-breakers codified in `state-buckets.md`** (deterministic when multiple buckets match): (1) uncommitted changes always wins, (2) open PR beats main-branch state, (3) long staleness beats recent shipping, (4) no `CLAUDE.md` always escalates to bucket D.
- **What `/code-health-advice` deliberately doesn't cover** (also in `state-buckets.md`): active debugging/incident response, greenfield feature work (that's `/plan-feature`), doc-only sessions (that's `/update-docs`). The advisor's job is the gray zone: "I have time, the repo is in some state, what's the best use of my next hour?"
- **Mid-session extension — `/code-cleanup --vulns` flag.** User asked "are we missing any aspect of code health across the 5 review skills?" Audit found dep CVE scanning was the strongest gap: `/code-cleanup --deps` finds *unused* deps but nothing audits installed deps for known CVEs. Picked a flag on existing skill over a new `/audit-deps` skill — same scan path (audit the deps tree), zero new carrying cost.
- **Three design choices for `--vulns`.** (1) **Opt-in only.** Default scans never call `npm audit` / `pip-audit` / etc. — those hit network registries, can fail offline, and slow the audit. The skill mentions the flag once; user opts in. (2) **Multi-stack coverage** — Node (npm/yarn/pnpm picked by lockfile), Python (pip-audit preferred, safety fallback), Rust (`cargo audit`), Go (`govulncheck`), PHP (`composer audit`), Ruby (`bundle audit`). Tool-missing case emits an "install hint" finding rather than silently emitting zero. (3) **Never auto-fixed by `--fix`.** Version bumps can break the app, transitive constraints get rewritten, and lockfile churn is project policy. Vulnerable deps are report-only — user runs the suggested `fix_command` themselves. Severity → risk mapping: critical/high → safe (high-confidence finding), moderate → likely_safe, low → needs_investigation.

Skipped as too stack-specific or wrong-tool: a11y, i18n, bundle size, license compliance, API contract checks.
